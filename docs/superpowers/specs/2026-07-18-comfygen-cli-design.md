# ComfyGen — CLI-помощник для генерации в ComfyUI

## Назначение

Небольшая Python-программа для терминала, которая проводит пользователя через путь «что хочу сгенерировать → какая модель потянет моё железо → какой промпт → готовый workflow в ComfyUI», не требуя от пользователя знания node-графов ComfyUI и характеристик собственного устройства.

Целевые платформы: **macOS** (основная), **Windows** (обязательна поддержка).

## Поток выполнения

```
run.sh / run.bat  (bootstrap)
  → проверяет venv, ставит зависимости при необходимости, запускает comfygen
      │
      ▼
1. Тип генерации: Фото / Видео / Аудио / Оживление фото (image-to-video)
2. Определение характеристик устройства — пользователь выбирает способ:
     a) через API запущенного ComfyUI (GET /system_stats)
     b) через ОС напрямую (psutil/platform, system_profiler / nvidia-smi / WMI)
     c) гибрид: API, а если Comfy не запущен — fallback на способ (b)
3. Загрузка каталога шаблонов Comfy (index.json), с локальным кешем (TTL ~24ч, флаг --refresh)
4. Фильтрация подходящих моделей/шаблонов по VRAM (Safe / Warning / Hidden)
5. Пользователь выбирает конкретную модель/шаблон из отфильтрованного списка
6. Ввод промпта; опционально — дополнительные параметры (negative prompt, resolution, seed, steps)
7. Для видео/оживления фото — либо путь к изображению сейчас, либо выбор позже в самом ComfyUI
8. Кастомизация workflow JSON (подстановка введённых значений в соответствующие узлы)
9. Сохранение через POST /userdata/workflows/<file>.json в запущенный ComfyUI —
   файл появляется в панели Workflows интерфейса, пользователь открывает и жмёт Queue
```

## Источник данных о моделях

Официальный репозиторий **Comfy-Org/workflow_templates**:

- `templates/index.json` — реестр из ~560 официальных шаблонов, сгруппированных по категориям (`image`, `video`, `audio`, `3d`). У 539 из них уже есть поле **`vram`** (в байтах) — это официальная метрика минимальных требований к видеопамяти от самого Comfy, а также `size` (объём скачивания), `models` (имена моделей), `tags` (`"Text to Image"`, `"Image to Video"` и т.п.).
- `templates/<name>.json` — сами workflow-файлы в **UI-формате** (nodes/links/groups — тот же формат, что при drag-n-drop импорте в интерфейс ComfyUI).

Самодельная справочная таблица VRAM по семействам моделей **не нужна** — используем готовые данные Comfy. Если у какого-то шаблона нет поля `vram`, такой шаблон помечается как «неизвестные требования» и не участвует в автофильтрации (показывается отдельно, по явному запросу пользователя).

«Оживление фото» реализуется как подмножество категории `video` с тегом `"Image to Video"`.

## Определение характеристик устройства

Три способа, выбор — на усмотрение пользователя при каждом запуске:

1. **API ComfyUI** (`GET /system_stats`) — точные данные (VRAM, device type cuda/mps/cpu, свободная память), но требует уже запущенного ComfyUI.
2. **ОС напрямую** — `psutil`/`platform` + платформенные вызовы:
   - macOS: `system_profiler SPDisplaysDataType` (Apple Silicon → unified memory)
   - Windows: `nvidia-smi` (NVIDIA) / WMI (общий случай)
   - Работает без запущенного ComfyUI.
3. **Гибрид** — сначала пробуем (1), при недоступности Comfy — fallback на (2).

**Apple Silicon (unified memory):** «доступный VRAM» считается не как весь объём RAM, а с эмпирическим запасом — например `total_RAM × 0.7`, т.к. память используется совместно системой и GPU. Точный коэффициент — предмет калибровки на этапе реализации/тестирования.

## Фильтрация моделей по VRAM

Сравниваем поле `vram` шаблона (байты) с обнаруженным объёмом видеопамяти устройства:

| Тир | Условие | Поведение |
|---|---|---|
| ✅ Safe | `vram_required ≤ available × 0.9` | Показывается как основной вариант |
| ⚠️ Warning | `available × 0.9 < vram_required ≤ available × 1.15` | Показывается с пометкой «может быть медленно / на грани, риск зависания» |
| 🚫 Hidden | `vram_required > available × 1.15` | Не предлагается — гарантированный OOM |

Коэффициенты `0.9`/`1.15` — отправная точка, выносятся в конфиг для последующей калибровки.

## Кастомизация workflow

- **Промпт**: поиск текстового узла (positive CLIPTextEncode-подобный, по имени/типу узла) в выбранном шаблоне, подстановка введённого пользователем текста.
- **Дополнительные параметры** (опционально, по явному запросу пользователя «настроить ещё что-то?»):
  - negative prompt (если у шаблона есть соответствующий узел)
  - resolution (узел EmptyLatentImage/аналог)
  - seed, steps (узел KSampler/аналог)
  Если пользователь отказывается — значения остаются как в оригинальном шаблоне Comfy.
- **Входное изображение** (для видео/оживления фото): пользователь выбирает —
  - указать путь сейчас → скрипт заливает файл через `POST /upload/image` и подставляет имя в узел `LoadImage`;
  - либо пропустить → узел `LoadImage` остаётся как в шаблоне, пользователь выбирает файл прямо в интерфейсе ComfyUI после открытия workflow (родная функциональность узла).

## Сохранение результата

Единственный сценарий (по решению пользователя): `POST /userdata/workflows/<file>.json` в запущенный ComfyUI. Файл сразу появляется в панели «Workflows» интерфейса; пользователю остаётся открыть его и нажать Queue.

Требует запущенного ComfyUI на момент сохранения. Если Comfy не отвечает — скрипт явно сообщает об этом и предлагает запустить ComfyUI и повторить попытку (без автоматического запуска самого ComfyUI — это вне зоны ответственности скрипта).

## Установка и запуск

- Обычный Python-пакет (`pyproject.toml`), но с self-bootstrapping launcher-скриптами:
  - `run.sh` (macOS) и `run.bat` (Windows) — при первом запуске проверяют наличие venv, создают его при отсутствии, ставят зависимости, затем запускают программу.
  - Пользователь не должен вручную выполнять `pip install` — всё делает сам скрипт.
- Зависимости: `rich`/`questionary`(или `InquirerPy`) для красивого интерактивного CLI, `httpx`/`requests` для HTTP, `psutil` для системной информации.

## Ошибки и граничные случаи

- ComfyUI не запущен на шагах, где это обязательно (сохранение через `/userdata`, детекция через API) — понятное сообщение с инструкцией, без падения с трейсбеком.
- GitHub недоступен при первом запросе `index.json` и локального кеша ещё нет — понятное сообщение об ошибке сети.
- Устройство не подходит ни под один шаблон выбранного типа генерации (даже Warning-тир) — сообщить об этом прямо и предложить сменить тип генерации.
- Шаблон без поля `vram` — не участвует в автофильтрации, доступен только через отдельный пункт «показать шаблоны с неизвестными требованиями».

## Тестирование

- Юнит-тесты на чистую логику: фильтрация по VRAM-тирам, поиск/подмена узлов в workflow JSON, парсинг `index.json`.
- Тесты HTTP-клиента к ComfyUI с mock-сервером (system_stats, userdata, upload/image).
- Ручной smoke-test на реальном ComfyUI: macOS (Apple Silicon) и Windows (NVIDIA), на каждом — минимум по одному сценарию для каждого из 4 типов генерации.
